# cobweb 设计提示词

> **用途**:在已有 pure CSS 自有设计系统(色彩与字体已定)的前提下,向设计阶段交代项目目标、用户画像和分模块的 UI/UX 优先级。
> **目标读者**:Claude Design / 设计师。
> **不在这份文档里**:具体色值、具体字体名 —— 由设计系统决定;以及具体技术实现 —— 见 `tech-stack.md`。

---

## 一、项目定位

cobweb 是基于 EasyTier 搭建的**私网管理中心**,与 ethub 默认同机部署。

它解决两件事:

1. **看清楚私网当前的状态** —— 节点连通性、链路质量、流量、服务日志、端口暴露
2. **从一个地方对所有节点下发改动** —— 配置、信任根、命令脚本等(可扩展)

它不是:
- 不是给团队用的多租户后台(单人使用)
- 不是替代 Grafana 的通用监控平台(只服务 mesh 拓扑这一种特定形态)
- 不是 K8s dashboard 那种重型管理面(节点数量是 10 量级,不是 1000)

---

## 二、唯一用户

**只有一个用户:项目作者本人。**

由此推导的设计含义:

- 不需要 onboarding、引导、提示气泡 —— 用户对每个按钮的含义有 ground truth
- 不需要权限模型、协作、评论
- 默认信息密度偏高 —— 用户偏好"一屏看到更多",而不是大留白
- 默认深色主题 —— 用户长时间盯屏
- 不需要 mobile responsive(但 tablet 横屏要 work,因为可能从 iPad 临时查看)
- 键盘操作优先级 ≥ 鼠标 affordance
- 错误信息直接显示原始 stderr / exit code,**不做"友好化"翻译** —— 用户能读懂技术细节,翻译反而损失信息

---

## 三、视觉氛围

一句话:**terminal-adjacent,professional,restrained**。

气质上靠近 SRE / 工程工具,不是 SaaS landing 风,也不是 cyberpunk 风。安静、密集、装饰为零、信号清晰。

### 正向参考

- **Tailscale admin** —— 最接近的场景类比:私网节点管理 + 信息密度 + 克制配色
- **Linear 设置页** —— 高密度信息 + 深色 + 几乎无装饰
- **GitHub Actions 矩阵视图** —— job × step 二维结果展示,状态色块
- **htop / k9s** —— TUI 工具的密度上限,可借鉴节奏(实际肯定比这两个柔和)

### 反向参考(明确不要)

- **Vercel / Netlify 的 dashboard** —— 视觉装饰过多、卡片、阴影、留白
- **Notion / Linear 工作区主视图** —— 卡片化、过大圆角,不适合矩阵密度
- **Apple HIG 风** —— 间距过大,信息密度太低

### 关键词

密集 · 克制 · 专业 · 信号清晰 · 装饰为零

---

## 四、反模式(快速排除)

- ❌ Hero 区 / 大留白 / 渐变背景 / illustration —— 这是工具,不是 landing page
- ❌ "Welcome to your dashboard!" 之类的欢迎文案
- ❌ 信息卡片化 / 圆角过大 —— 降低密度,不适合矩阵视觉
- ❌ 把 stderr 翻译成"Oops! Something went wrong" —— 直接展示原文
- ❌ 浮动 toast 通知重要信息 —— 用户会错过;用 in-place 状态变化
- ❌ Modal 套 Modal
- ❌ 隐藏功能在三级菜单 / 折叠面板深处
- ❌ 动画过渡 > 200ms
- ❌ 全屏 spinner —— 用 shimmer / pulse 表达 loading,且只作用于"正在变化的元素"

## 五、颜色调性需求

> 具体色值由设计系统决定。这里只说**需要哪些 semantic 角色 + 调性方向**。

### 主调:中性灰为主,状态色作为信号

整体氛围应当"安静":大部分时间屏幕是灰阶的,状态色不滥用 —— **正因为不滥用,出问题时颜色才能跳出来成为信号**。

深色主题为默认(具体明度由设计系统定)。

### 必需的 semantic 状态色

设计系统至少需提供:

| 角色 | 语义 | 调性方向 |
|---|---|---|
| `ok` | 正常 / 连通 / 已应用 | 冷色系倾向(绿 / 青) |
| `warn` | 降级 / 部分成功 / 高延迟 | 暖色系倾向(琥珀) |
| `error` | 不可达 / 失败 | 红 / 品红系 |
| `unknown` | 未知 / 未配置 / 已忽略 | 中性灰 |
| `loading` / `in-progress` | 任务执行中 | 蓝 / 青系倾向(与 ok 可区分) |
| `stale` | 数据陈旧 | 同 ok 但降饱和 / 提透明度 |
| `not-managed` | 节点存在但不纳入管理 | 同 unknown 但更弱 |

### 链路质量梯度(矩阵 cell 背景核心 token)

需要一组从 `ok` 到 `error` 的**连续色阶**(4–6 档),用于:
- 矩阵 cell 按"延迟从低到高"或"丢包从 0 到高"上色
- 时间序列图的阈值带

---

## 六、Typography 需求

> 具体字体由设计系统决定。这里只说需要哪些字族角色 + 字号下限。

### 字族角色

- **主字体** —— UI 文案、表单、导航
- **monospace** —— 节点名、IP、hostname、日志、shell 输出、命令、stderr、矩阵 cell 内的数字

### 字号下限

- 主体 UI 字号下限 **12px**(高密度场景)
- 矩阵 cell 内的数字可下到 **11px**
- 不要更小

---

## 七、动效原则

**几乎无动效**。

允许的两处:
- **hover 状态切换** —— ≤150ms,subtle(背景、边框色变化)
- **数据更新** —— cell 数值变化时 ≤150ms 渐变,而非硬切

明确禁止:
- 入场动画 / 滑入 / 淡入
- 全屏 spinner(loading 用 shimmer 或 pulse 表达,只作用于"正在变化的元素")
- 任何 > 200ms 的过渡

---

## 八、全局 UX 原则

1. **状态颜色编码统一** —— 全产品一致,见第五节
2. **数据陈旧度可见** —— 每个实时模块右上角显示"上次更新于 X 秒前",超阈值变 `stale` 色
3. **操作的可中断 / 可重试是默认能力**,不是高级功能 —— 失败的子任务有显眼的"重试这一个"按钮
4. **日志 / 输出永远完整展示** —— 不裁剪。需要的话 monospace + 可折叠区域,但**不要省略号 + "更多"按钮**
5. **不用 modal 做主要操作** —— modal 仅用于:危险操作确认、< 5 字段的短表单;复杂表单走专用页面或侧边面板
6. **路由直接对应模块** —— URL 是可分享的状态,深链接到具体节点、具体任务详情
7. **节点状态是二维的:`mesh × agent`** —— 一个节点可能"在 mesh 内但 agent 离线"(降级:只有延迟矩阵 + SSH 兜底),或"agent 在线但 mesh 短暂抖动"。UI 上**两个维度要同时表达**,不要折叠成单一的"在线/离线"

## 九、模块一:Dashboard(监控大盘)

### 主视图:节点矩阵

**N×N 矩阵是这个模块的核心隐喻**。行/列都是节点,cell 表示一对节点之间的链路质量。

**视觉路线:数据网格 + 状态色,不是纯热力图。**
- 纯热力图(只有颜色块)会丢失精确数字 —— 工程师场景不接受
- 应该是:**背景按状态/梯度上色,前景显示数字**;数字小但清晰可读,背景是主信号

#### 设计优先级

| 优先级 | 项 |
|---|---|
| P0 | cell 一眼看出"通 / 不通 / 降级",颜色优先于数字 |
| P0 | 节点名(hostname)在行头列头完整可读,不截断 |
| P0 | cell hover 显示完整 metrics:延迟、丢包、流量、cost、tunnel proto |
| P1 | 节点 ≤ 12 时一屏放得下;> 12 时滚动,行列头 sticky |
| P1 | cell click → 钻取这对节点的时间序列图(延迟、丢包曲线) |
| P1 | 矩阵上方"指标切换"—— 显示延迟 / 丢包 / 流量速度;切换时矩阵布局不变,只改色与值 |
| P2 | 对角线(自身 → 自身)灰色或斜杠,不参与配色 |
| P2 | 链路方向性 toggle:A→B 与 B→A 不一定相等,初版只显示一个方向 |

#### 视觉规则

- Cell 正方形,推荐 56–72px
- Cell 内信息分层:背景色(状态梯度) + 主数字(当前指标) + 可选角标(趋势箭头)
- Cell **不放图标**(降低信息密度)
- Cell 之间 1px 间隙(深色背景下可见)

### 子视图:节点详情

矩阵点击节点行/列头进入。包含:

| 区块 | 内容 | 优先级 |
|---|---|---|
| 基本信息 | hostname、ipv4、os、版本、cost、tunnel proto | P0 |
| Agent 状态 | agent version、connected since、last heartbeat、断开重连按钮 | P0 |
| 连通的 peers | 该节点视角的 peer 列表 + 各 peer 的延迟丢包 | P0 |
| 服务日志 | EasyTier 服务的 stdout/stderr 实时 tail | P0 |
| 端口开放情况 | `ss -tlnp` / `netstat` 结果,标注 EasyTier 占用的端口 | P1 |
| 时间序列图 | 该节点的 rx/tx 速度、cpu/mem(如可获得) | P1 |
| 快速 SSH | "Open SSH" 按钮,在面板内或新页签打开 terminal | P1 |

### 子模块:SSH 快速连接

- **入口**:节点详情页 + 矩阵 cell 右键菜单
- **形态**:浏览器内 terminal(xterm.js + WebSocket + PTY)
- **通道**:优先走 agent 的 exec API(NAT 后节点也能开);agent 离线时自动降级 SSH 兜底。**用户感知一致,UI 不暴露通道差异**(底部状态条可选标注"via agent / via ssh")
- **定位**:**不是替代 iTerm / Windows Terminal**,而是"快速看一眼 / 改一行配置"
- **生命周期**:同时最多 ~4 个会话,各自独立 tab;关浏览器即断,不做会话恢复

#### 设计优先级

| 优先级 | 项 |
|---|---|
| P0 | 等宽字体、深色背景、可粘贴 |
| P0 | 顶部显示连接的节点名 + user@host;断开有明显视觉提示 |
| P1 | 支持 Ctrl+C / Ctrl+D 等常用按键透传 |
| P2 | 多 tab 切换 |

### Dashboard 整体布局参考

```
┌────────────────────────────────────────────────────────────┐
│ 顶部导航:Dashboard | 分发中心          右侧:更新时间       │
├────────────────────────────────────────────────────────────┤
│ ┌─指标切换─────┐  ┌─总览数字───────────────────────────┐  │
│ │延迟│丢包│流量│  │节点 N · 在线 M · 异常链路 K         │  │
│ └──────────────┘  └──────────────────────────────────────┘  │
│                                                              │
│              N × N 节点矩阵(主区域)                         │
│                                                              │
│ ┌─选中节点详情侧栏(可折叠)──────────────────────────┐    │
│ │ hostname / peers / 日志 tail / 端口...               │    │
│ └─────────────────────────────────────────────────────┘    │

```

## 十、模块二:分发中心

### 心智模型

**这是一个"任务执行器",不是"配置管理器"**。

每次下发是一次 task:**选择目标节点 → 选择能力(kind) → 填参数 → 执行 → 看结果**。任务有状态、有历史、可重试。

当前已有的能力:
- **DNS 设置** —— 写入 systemd-resolved / `/etc/resolver` / NRPT
- **CA 信任根分发** —— 从 kubectl 拉,推到各节点 trust store
- **SSH key 管理 + mesh ssh 配置** —— 私钥分发 + authorized_keys + ssh config fence

未来扩展点(留位置即可,不需要现在出 UI):
- **Agent 安装 / 升级 / 卸载** —— SSH 通道执行,把 Rust agent 二进制推送到节点并注册为系统服务
- **通用脚本分发**(任意 shell / powershell)—— 优先走 agent exec,SSH 兜底
- **通用文件分发**(任意 src → 任意 dst + 安装命令)—— 优先走 agent file_put,SSH 兜底
- **EasyTier 配置同步** —— 通过 agent 调本机 `easytier-cli` 修改,reload 服务
- 证书 / 密钥轮换

### 设计优先级

| 优先级 | 项 |
|---|---|
| P0 | 左侧导航能力列表,右侧能力操作面板。新增能力是"加一项",不是"改布局" |
| P0 | 每次执行后展示结果矩阵:**行是目标节点,列是步骤**(probe / upload / install / verify),格子展示成功 / 失败 / 跳过 |
| P0 | 失败格子点击展开 stderr + 失败的具体命令,**完整不截断** |
| P0 | "重试失败节点"按钮 —— 只对失败的节点重跑,不动成功的 |
| P1 | 任务历史:列出过去 N 次执行,可点开看当时的结果矩阵 |
| P1 | "Dry run" 模式:展示将要执行的命令,但不实际执行 |
| P2 | 任务执行中实时显示每个节点当前在哪一步(SSE 推送) |
| P2 | "保存为预设":如"上线一台新机器" = ssh setup + ca install + dns config 串联 |

### 通用任务结果矩阵(核心组件)

无论 mesh / dns / ca apply,**结果统一用同一种 UI 呈现**,因为底层都是"在 N 个节点上执行 K 个步骤":

```
节点 \ 步骤   │ probe │ upload │ install │ verify
─────────────┼───────┼────────┼─────────┼───────
archmbp      │   ✓   │   ✓    │    ✓    │   ✓
nuc-arch     │   ✓   │   ✓    │    ✗    │   -
work-mbp     │   ✓   │   -    │    -    │   -    │ (skipped)
aliyun       │   ✗   │   -    │    -    │   -
```

视觉类比:**GitHub Actions 的 job × step 矩阵**,但 cell 是颜色块 + 状态符号,不是大 icon。

点击 `✗` 展开完整错误详情(stderr + 命令)。

### 单节点选择 UI

- 默认全选 managed 节点
- 顶部:全选 / 全不选 / 反选 工具栏
- 节点 chip 显示**复合状态**:`mesh 在线` + `agent 在线` 两个独立信号点(参考第八节原则 7)。可能的组合:
  - mesh ✓ agent ✓ —— 全功能可用
  - mesh ✓ agent ✗ —— 降级:只能走 SSH 通道执行任务
  - mesh ✗ agent ✓ —— 罕见但可能(mesh 抖动);仍可执行 agent 范围内的任务
  - mesh ✗ agent ✗ —— 不可达;**视觉降级但不禁用**(用户可能就是要"先准备好,等它上线")

### 扩展点的视觉传达

新加分发能力时,设计应让"这是一种新能力"显而易见,而不是"这是一个特殊页面":
- 左侧导航:能力 = 列表项,小图标 + 名称 + 简短描述
- 右侧:能力的参数 schema → 自动生成表单
- 设计系统需提供通用的:`FormField`(text / select / multi-select / toggle / textarea / file)、`NodePicker`、`ParamGroup`

## 十二、设计系统需要交付的组件清单(工程参考)

> 这一节为工程实现侧的需要列出,**设计师不必照此组织设计稿**。但产物最终落到组件级别时,这些是必要的。

按出现频率排:

1. `StatusDot` —— 圆点 + 颜色 + 可选 tooltip
2. `MetricCell` —— 矩阵单元,接受 value + status + tooltip
3. `NodeChip` —— 节点名 + 在线状态点 + 可点击
4. `TaskResultMatrix` —— 节点 × 步骤的结果矩阵(分发中心专用)
5. `LogStream` —— monospace + auto-scroll + 可暂停 + 可搜索
6. `SSHTerminal` —— xterm.js 封装
7. `TimeSeriesChart` —— 折线图
8. `NodePicker` —— 多选节点
9. `FormField` —— 通用表单字段(text / select / multi-select / toggle / textarea / file)
10. `Panel` / `Section` —— 内容容器(保持高密度,不要过度内边距)
